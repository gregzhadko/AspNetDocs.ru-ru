## <a id="_TOC3"></a>  Новые возможности

В этом разделе описаны возможности, которые были введены в версии MVC 2.

### <a id="_TOC3_1"></a>  Шаблонизированные вспомогательные объекты

Шаблонизированные вспомогательные объекты позволяют автоматически связать элементы HTML для редактирования и отображения с типами данных. Например при отображении в виде данных типа System.DateTime элемент пользовательского интерфейса выбора даты могут автоматически подготавливаться. Это похоже на работу шаблоны полей, используемых в динамических данных ASP.NET. Дополнительные сведения см. в разделе [использование шаблонизированных вспомогательных объектов для отображения данных](https://go.microsoft.com/fwlink/?LinkId=159062) на сайте MSDN.

### <a id="_TOC3_2"></a>  Области

Области позволяют организовать большого проекта на несколько секций меньшего размера, чтобы справляться со сложностями, большие веб-приложения. Каждый раздел («область») обычно представляет отдельный раздел больших веб-узла и используется для группировки связанных наборов контроллеров и представлений. Дополнительные сведения см. в разделе [Пошаговое руководство: Упорядочение приложения ASP.NET MVC, областям](https://go.microsoft.com/fwlink/?LinkId=158978) на сайте MSDN.

### <a id="_TOC3_6"></a>  ModelMetadata и ModelMetadataProvider классы

Класс ModelMetadataProvider предоставляет абстракцию для получения метаданных для модели в представлении. MVC 2 включает в себя поставщик по умолчанию, который делает доступными метаданных, предоставляемого через атрибуты в пространстве имен System.ComponentModel.DataAnnotations. Существует возможность создания метаданных поставщиков, которые содержат метаданные из других хранилищ данных, таких как базы данных или файлы XML.

Класс ViewDataDictionary предоставляет объект ModelMetadata, который содержит метаданные, которые извлекаются из модели с помощью класса ModelMetadataProvider. Это позволяет шаблонизированных вспомогательных методов использовать эти метаданные и соответствующим образом изменить свои выходные данные.

Дополнительные сведения см. в документации по [ModelMetadata](https://msdn.microsoft.com/library/system.web.mvc.modelmetadataprovider(VS.100).aspx) и [ModelMetadataProvider](https://msdn.microsoft.com/library/system.web.mvc.modelmetadataprovider(VS.100).aspx) классы.

ASP.NET MVC 2 поддерживает использование RangeAttribute, RequiredAttribute, StringLengthAttribute и RegexAttribute атрибуты проверки (определенный в пространстве имен System.ComponentModel.DataAnnotations) при привязке к модели, чтобы предоставить входные данные Проверка.

Дополнительные сведения см. в разделе [Как Проверка с помощью атрибутов DataAnnotations модели данных](https://go.microsoft.com/fwlink/?LinkId=159063) на сайте MSDN. ### <a id="_TOC3_8"></a>  Поставщиков средств проверки модели

Класс поставщика проверки модели представляет собой абстракцию, который предоставляет логику проверки для модели. ASP.NET MVC включает в себя поставщик по умолчанию, исходя из атрибутов проверки, включенные в пространство имен System.ComponentModel.DataAnnotations. Можно также создать собственные поставщики проверки, которые определяют пользовательские правила проверки и пользовательские сопоставления правил проверки для модели. Дополнительные сведения см. в документации по [ModelValidatorProvider](https://msdn.microsoft.com/library/system.web.mvc.ModelValidatorProvider(VS.100).aspx) класса.

### <a id="_TOC3_9"></a>  Проверка на стороне клиента

Класс поставщика средства проверки модели предоставляет метаданные проверки в браузере в виде JSON-сериализованных данных, которые могут использоваться библиотекой проверки на стороне клиента. ASP.NET MVC 2 включает клиентскую библиотеку проверки и адаптер, поддерживающий DataAnnotations атрибуты проверки пространства имен, записанные ранее. Класс поставщика также позволяет использовать другие библиотеки клиентской проверки, написав адаптер, который обрабатывает данные JSON и выполняет вызов в альтернативной библиотеки.

Переопределение может использоваться только в тех случаях, когда реального запроса является запрос POST. Значение переопределения игнорируются для запросов, использующих любой другой глагол HTTP.

### <a id="_TOC3_13"></a>  Новый класс HiddenInputAttribute для шаблонизированных вспомогательных методов

Новый атрибут HiddenInputAttribute можно применить к свойству модели, чтобы указать, следует ли отрисовывать скрытый элемент input, при отображении модели в шаблоне редактора. (Атрибут задает неявное значение UIHint HiddenInput). Свойства атрибута DisplayValue позволяет указать, отображается ли значение в редакторе и режимы отображения. Когда DisplayValue имеет значение false, ничего не отображается, даже HTML-разметка, которая обычно окружает поля. Значение по умолчанию для DisplayValue имеет значение true.

Можно использовать атрибут HiddenInputAttribute в следующих сценариях:

- Когда представление позволяет изменить идентификатор объекта и это необходимо для отображения значения также программного обеспечения скрытый элемент input, содержащий старый идентификатор, чтобы ее можно передать обратно к контроллеру.
- Если представление позволяет пользователям изменить двоичное свойство, которое никогда не отображать, такие как свойство метки времени. В этом случае значение и окружающей разметкой HTML (например, метки и значения) не отображаются.

Если атрибут установлен в значение true (или параметр не указан), происходит следующее:

- В редакторе шаблонов метка выводится и значение отображается в скрытом элементе ввода.

Когда атрибут имеет значение false, происходит следующее:

- В редакторе шаблонов метка не отображается, и значение отображается в скрытом элементе ввода.

В Visual Studio 2010 значение равно .NET 3.5 или .NET 4.

## <a id="_TOC4"></a>  Улучшения API

В этом разделе описаны изменения в существующие типы ASP.NET MVC и элементов.

- Добавлен защищенный виртуальный метод CreateActionInvoker в классе контроллера. Этот метод вызывается свойство ActionInvoker контроллера и позволяет для отложенное создание экземпляра средства вызова, если нет вызова уже задано.
- Добавлен защищенный виртуальный метод HandleUnauthorizedRequest в классе AuthorizeAttribute. Это позволяет фильтры, которые являются производными от AuthorizeAttribute для управления поведением при сбое авторизации.

## <a id="_TOC5"></a>  Критические изменения

Следующие изменения могут привести к ошибкам в существующие приложения ASP.NET MVC 1.0.

#### <a name="change-in-property-validation-behavior-for-classes-that-implement-idataerrorinfo"></a>Изменение в поведении проверки свойств для классов, реализующих IDataErrorInfo

Для объектов модели, использующие IDataErrorInfo для проверки проверяется каждое свойство, независимо от того, определить, было ли задано новое значение. В ASP.NET MVC 1.0 были проверены только те свойства, которые бы задать новые значения. В ASP.NET MVC 2 свойство ошибка IDataErrorInfo вызывается только в том случае, если все проверяющие элементы управления свойство выполнены успешно.

#### <a name="iis-script-mapping-script-is-no-longer-available-in-the-installer"></a>Сценарий сопоставления скриптов IIS больше не доступен в установщике

Сценарий сопоставления скриптов IIS — это сценарий командной строки, которая используется для настройки сопоставления скриптов IIS 6 и IIS 7 в классическом режиме. Скрипт сопоставление сценария не требуется при использовании Visual Studio Development Server, или при использовании IIS 7 в интегрированном режиме. Сценарии доступны в качестве отдельной загрузки не поддерживается на [Интернет-стек любви ASP.NET](https://github.com/aspnet/AspNetWebStack).

#### <a name="the-htmlsubstitute-helper-method-in-mvc-futures-is-no-longer-available"></a>Вспомогательный метод Html.Substitute в MVC Futures больше не доступен

Из-за изменений в поведении отрисовки обработчиков представлений MVC Html.Substitute вспомогательный метод не работает и был удален.

#### <a name="the-ivalueprovider-interface-replaces-all-uses-of-idictionary"></a>Интерфейс IValueProvider заменяет все случаи использования IDictionary

Каждого аргумента свойство или метод, который принимается IDictionary в MVC 1.0 теперь принимает IValueProvider. Это изменение затрагивает только приложения, которые включают пользовательские поставщики значений или пользовательские связыватели модели. Следующие примеры свойств и методов, на которые влияет это изменение.

- Свойства ValueProvider ControllerBase и ModelBindingContext классов.
- TryUpdateModel методы класса контроллера.

#### <a name="new-css-classes-were-added-in-the-sitecss-file"></a>Были добавлены новые классы CSS в файле Site.css

Файл Site.css в шаблонах проектов ASP.NET MVC был обновлен для включения новых стилей, используемых функциональных возможностей проверки и шаблонизированных вспомогательных методов.

#### <a name="helpers-now-return-an-mvchtmlstring-object"></a>Вспомогательные функции теперь возвращают объект MvcHtmlString

Чтобы воспользоваться преимуществами новых HTML-кодирование синтаксиса выражения в ASP.NET 4, возвращаемый тип для вспомогательных методов HTML — теперь MvcHtmlString вместо строки. При использовании ASP.NET MVC 2 и новых вспомогательных функций в ASP.NET 3.5, вы не сможете воспользоваться преимуществами синтаксиса HTML-кодирование; новый синтаксис доступен только при запуске ASP.NET MVC 2 в ASP.NET 4.

#### <a name="jsonresult-now-responds-only-to-http-post-requests"></a>JsonResult теперь отвечает только на запросы HTTP POST

Для устранения JSON чужого, по умолчанию, которые могут раскрытие информации, класс JsonResult теперь отвечает только на запросы HTTP POST. ПОЛУЧИТЬ AJAX вызовы к методам действий, которые возвращают объект JsonResult следует изменить на вместо этого используйте POST. При необходимости можно переопределить это поведение, задав новое свойство JsonRequestBehavior JsonResult. Дополнительные сведения о потенциальных уязвимостей, см. в записи блога [захват JSON](http://haacked.com/archive/2009/06/25/json-hijacking.aspx) в блоге Фила Хаака.

#### <a name="model-and-modeltype-property-setters-on-modelbindingcontext-are-obsolete"></a>Модели и методы задания свойств ModelType на ModelBindingContext являются устаревшими

Новое свойство настраиваемым ModelMetadata добавлен к классу ModelBindingContext. Новое свойство инкапсулирует модель и ModelType свойства. Несмотря на то, что свойства модели и ModelType являются устаревшими, для обеспечения обратной совместимости методы получения свойств по-прежнему работать; они делегировать свойство ModelMetadata для получения значения.

#### <a name="changes-to-the-defaultcontrollerfactory-class-break-custom-controller-factories-that-derive-from-it"></a>Изменения в класс DefaultControllerFactory нарушать работу фабрик пользовательского устройства, которые являются производными от него

Класс DefaultControllerFactory была исправлена, удалив свойство RequestContext. Вместо этого свойства защищенных виртуальных методов GetControllerInstance и GetControllerType передается экземпляр контекста запроса. Это изменение затрагивает настраиваемый контроллер фабрики, которые являются производными от DefaultControllerFactory.

Настраиваемый контроллер фабрики часто используются для предоставления приложениям возможность встраивания зависимостей для ASP.NET MVC. Чтобы обновить настраиваемый контроллер фабрики для поддержки ASP.NET MVC 2, измените сигнатуру метода или подписей в соответствии с новые подписи и использовать параметр контекста запроса вместо свойства.

#### <a name="area-is-a-now-a-reserved-route-value-key"></a>«Область» — теперь это зарезервированное значение маршрута ключ

Строка «область» в значениях маршрута теперь имеет особое значение в ASP.NET MVC, таким же образом, что «controller» и «действие». Один подразумевается, что если вспомогательные методы HTML входят в состав значение маршрута словарь, содержащий «область», вспомогательные функции больше не добавит «область» в строке запроса.

Если вы используете функцию области, убедитесь, что не использовать {области} как часть URL-адреса маршрута.
