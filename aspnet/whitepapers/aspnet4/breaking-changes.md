## <a name="browser-definition-files-updated"></a>Файлы описания браузеров обновлены

Файлы описания браузеров обновлены: теперь они включают сведения о новых и обновленных браузерах и устройствах. Устаревшие браузеры и устройства (например, Netscape Navigator) удалены; добавлены новые браузеры и устройства (например, Google Chrome и Apple iPhone).

Если ваше приложение содержит пользовательские описания браузеров, унаследованные от одного из удаленных описаний браузеров, произойдет ошибка. Например если `App_Browsers` папка содержит определения браузера, который наследует от определения браузера IE2, вы получите следующее сообщение об ошибке конфигурации:

- Не удается найти элемент обозревателя или шлюза с Идентификатором «IE2».

> [!NOTE]
> **HttpBrowserCapabilities** объекта (предоставляемый страницы **Request.Browser** свойство) определяется в файлах определения браузеров. ## <a name="systemwebmobiledll-removed-from-root-web-configuration-file"></a>System.Web.Mobile.dll, удалены из корневого файла веб-конфигурации

В предыдущих версиях ASP.NET, ссылку на сборку System.Web.Mobile.dll был включен в корневом каталоге `Web.config` файл **сборки** раздела. Чтобы повысить производительность, ссылку на эту сборку удалена.

Сборка System.Web.Mobile.dll включается в ASP.NET 4, но она устарела. Если вы хотите использовать типы из сборка System.Web.Mobile.dll, добавьте ссылку на эту сборку либо в корневой каталог `Web.config` файла или приложения `Web.config` файл. Например, если вы хотите использовать любой из (не рекомендуется) мобильных элементов управления ASP.NET, необходимо добавить ссылку на сборку System.Web.Mobile.dll на `Web.config` файл.

## <a name="aspnet-request-validation"></a>Проверка запросов ASP.NET

Функция проверки запросов в ASP.NET обеспечивает определенный уровень защиты по умолчанию от атаки межузловых сценариев (XSS). В предыдущих версиях ASP.NET проверка запроса была включена по умолчанию. Тем не менее, он применяется только к страницам ASP.NET (`.aspx` файлов и их файлы классов) и только при выполнении этих страниц.

В ASP.NET 4 по умолчанию включена проверка запросов для всех запросов, так как он используется перед **BeginRequest** этап HTTP-запроса. В результате проверка запросов применяется к запросы для всех ресурсов ASP.NET, а не только запросов страниц .aspx. Сюда входят запросы, такие как вызовы веб-службы и пользовательские обработчики HTTP. Проверка запроса уже активен при чтении содержимого запроса HTTP пользовательских HTTP-модулей.

## <a name="configuration-errors-related-to-new-aspnet-4-root-configuration"></a>Ошибки конфигурации, связанных с новой конфигурацией ASP.NET 4 корневой

Корневые файлы конфигурации ( `machine.config` файл и корнем `Web.config` файл) для .NET Framework 4 (и поэтому ASP.NET 4) были обновлены для включения большая часть стандартных сведений конфигурации, в ASP.NET 3.5 был найден в приложение `Web.config` файлов. Из-за сложности в управляемых системах конфигурации IIS 7 и IIS 7.5 запуск приложений ASP.NET 3.5 на платформе ASP.NET 4 и с IIS 7 или IIS 7.5 может привести ASP.NET или IIS ошибки конфигурации.

Рекомендуется обновить приложения ASP.NET 3.5 до ASP.NET 4 с помощью средства обновления проекта в Visual Studio 2010, если это возможно. Visual Studio 2010 автоматически изменяет приложения ASP.NET 3.5 `Web.config` файл, содержащий параметры для ASP.NET 4.

Тем не менее это поддерживаемый сценарий для запуска приложений ASP.NET 3.5, с помощью .NET Framework 4 без повторной компиляции. В этом случае может потребоваться вручную изменить приложения `Web.config` файл перед запуском приложения под .NET Framework 4 и IIS 7 или IIS 7.5.

В следующих двух разделах описываются изменения, которые может потребоваться внести для различных сочетаний программного обеспечения.

**С пакетом обновления 1 для Windows Vista или Windows Server 2008 с пакетом обновления 1, где установлены ни исправление KB958854 ни с пакетом обновления 2.** В этой конфигурации системы конфигурации IIS 7 неверно объединяет конфигурации управляемого приложения, сравнивая уровня приложения `Web.config` файлов в ASP.NET 2.0 `machine.config` файлов. Из-за этого, на уровне приложений `Web.config` файлы из .NET Framework 3.5 или более поздней версии должны иметь **system.web.extensions** определение раздела конфигурации (элемент), чтобы не вызвать ошибку проверки IIS 7.

Тем не менее, изменены вручную уровня приложения `Web.config` записи файлов, которые не соответствуют исходной стандартным определениям раздела конфигурации, которые появились в Visual Studio 2008 приводит к ошибкам конфигурации ASP.NET. (Записей конфигурации по умолчанию, созданные Visual Studio 2008 работать правильно.) Общая проблема заключается в изменении вручную `Web.config` оставлять файлы **allowDefinition** и **requirePermission** атрибуты конфигурации, которые находятся на различных раздела конфигурации определения. Это приводит к возникновению рассогласования раздела сокращенное настройки уровня приложения `Web.config` файлы и полное определение в ASP.NET 4 `machine.config` файл. Таким образом во время выполнения, система конфигурации ASP.NET 4 вызывает ошибку конфигурации.

**Windows Vista SP2, Windows Server 2008 SP2, Windows 7, Windows Server 2008 R2, а также Windows Vista с пакетом обновления 1 и Windows Server 2008 SP1, где установлено исправление KB958854.**

В этом случае система Конфигурация собственного IIS 7 и IIS 7.5 возвращает ошибку конфигурации, так как он выполняет сравнение текста на **тип** атрибут, который определен для обработчик раздела конфигурации управляемых. Так как все `Web.config` файлы, созданные в Visual Studio 2008 и Visual Studio 2008 SP1 имеют «3.5» в строке типа **system.web.extensions** (и связанных) обработчиков разделов конфигурации и поскольку ASP.NET 4 `machine.config` файл имеет «4.0» в **тип** атрибут для тех же обработчиков раздела конфигурации приложений, которые всегда создаются в Visual Studio 2008 или Visual Studio 2008 SP1 не проходят проверку конфигурации в IIS 7 и IIS 7.5.

### <a name="resolving-these-issues"></a>Для устранения подобных проблем

Существует обходной путь для первого сценария для обновления уровня приложения `Web.config` файла путем включения стандартного текста конфигурации из `Web.config` файл, который был создан автоматически с помощью Visual Studio 2008.

Второе решение для первого сценария — для установки с пакетом обновления 2 для ОС Vista или Windows Server 2008 на компьютере или установите исправление KB958854 ([https://support.microsoft.com/kb/958854](https://support.microsoft.com/kb/958854)) для исправления поведения неправильные конфигурации и объединения Системы конфигурации IIS. Однако после выполнения любого из этих действий, приложения могут возникнуть ошибки конфигурации из-за проблемы, описанные во втором сценарии.

Решение для второй сценарий — удалите или закомментируйте все **system.web.extensions** определения разделов конфигурации и раздел конфигурации группы определений из уровня приложения `Web.config` файл. Определения этих обычно находятся в верхней части уровня приложения `Web.config` файл и можно определить по **configSections** элемент и его дочерние элементы.

## <a name="the-httprequestfilepath-property-no-longer-includes-pathinfo-values"></a>Свойство HttpRequest.FilePath больше не включает в себя PathInfo значения

**HttpRequest.PathInfo**: `SomeAction`

## <a name="aspnet-20-applications-might-generate-httpexception-errors-that-reference-eurlaxd"></a>ASP.NET 2.0 приложения могут вызывать ошибки HttpException, которые ссылаются на eurl.axd

После включения ASP.NET 4 в IIS 6 приложения ASP.NET 2.0, работающие в IIS 6 (в Windows Server 2003 или Windows Server 2003 R2) могут вызывать ошибки наподобие следующих:

`System.Web.HttpException: Path '/[yourApplicationRoot]/eurl.axd/[Value]' was not found.`

Эта ошибка возникает, когда ASP.NET обнаруживает, что веб-сайт настроен для использования ASP.NET 4, собственный компонент ASP.NET 4 передает URL-адрес без расширений управляемую часть среды ASP.NET для дальнейшей обработки. Тем не менее если виртуальные каталоги, расположенные ниже 4 веб-сайта ASP.NET настроены для использования ASP.NET 2.0, обработка URL-адреса без расширений в результатах этого способа в измененный URL-адрес, содержащий строку «eurl.axd». Затем этот измененный URL-адрес отправляется в приложение ASP.NET 2.0. ASP.NET 2.0 не распознает формат «eurl.axd». Таким образом, ASP.NET 2.0 пытается найти файл с именем `eurl.axd` и выполните его. Так как такой файл не существует, происходит сбой запроса с **HttpException** исключение.

Можно обойти эту проблему, с помощью одного из следующих вариантов.

### <a name="option-1"></a>Вариант 1

Если ASP.NET 4 не требуется для запуска веб-сайта, измените сопоставление сайта для использования ASP.NET 2.0.

### <a name="option-2"></a>Вариант 2

Если ASP.NET 4 требуется для запуска веб-сайта, переместите любые дочерние виртуальные каталоги ASP.NET 2.0 на другой веб-сайт, который сопоставлен с ASP.NET 2.0.

### <a name="option-3"></a>Вариант 3

Если не рекомендуется для повторного сопоставления веб-узла ASP.NET 2.0 или изменение расположения виртуального каталога, явно отключите URL-адресов без обработки в ASP.NET 4. Используйте следующую процедуру:

1. В реестре Windows откройте следующий узел:

`HKEY_LOCAL_MACHINE\SOFTWARE\Microsoft\ASP.NET\4.0.30319.0`

1. Создайте новый **DWORD** значение с именем **EnableExtensionlessUrls**.
2. Задайте **EnableExtensionlessUrls** 0. Это приведет к отключению без расширений поведения URL-адрес.
3. Сохраните значение реестра и закройте редактор реестра.
4. Запустите **iisreset** средство командной строки, что приводит к IIS для чтения новое значение реестра.

> [!NOTE]
> Установка **EnableExtensionlessUrls** 1 включает без расширений поведение URL-адрес. Это параметр по умолчанию, если значение не указано.

## <a name="event-handlers-might-not-be-not-raised-in-a-default-document-in-iis-7-or-iis-75-integrated-mode"></a>Обработчики событий могут не не вызываться в документе по умолчанию в IIS 7 или IIS 7.5 режим интеграции

ASP.NET 4 включает в себя изменения, которые изменяют способ **действие** атрибут HTML- **формы** элемент отображается в том случае, когда URL-адрес без расширений преобразует документ по умолчанию. Пример URL-адрес без расширений, разрешения на документ по умолчанию будет [ http://contoso.com/ ](http://contoso.com/), в запрос на результате [ http://contoso.com/Default.aspx ](http://contoso.com/Default.aspx).

ASP.NET 4 теперь отображает HTML- **формы** элемента **действие** значение атрибута как пустая строка, при выполнении запроса к URL-адрес без расширения, сопоставленного с ним документ по умолчанию. Например, в более ранних выпусках ASP.NET запрос к [ http://contoso.com ](http://contoso.com) приведет к запрос на `Default.aspx`. В этом документе открывающий **формы** тег отображался бы, как показано в следующем примере:

`<form action="Default.aspx" />`

В ASP.NET 4 запрос на [ http://contoso.com ](http://contoso.com) также приводит к запросу к `Default.aspx`. Тем не менее, ASP.NET теперь отображает открывающий HTML **формы** тег, как показано в следующем примере:

`<form action="" />`

Это различие в способе **действие** визуализируется атрибут может вызвать незначительные изменения в обработке формы post, IIS и ASP.NET. Когда **действие** атрибута является пустой строкой, IIS **DefaultDocumentModule** объект создаст дочерний запрос к `Default.aspx`. В большинстве случаев этот дочерний запрос прозрачен для кода приложения и `Default.aspx` страница выполняется в обычном режиме.

Однако возможное взаимодействие между управляемым кодом и интегрированным режимом IIS 7 или IIS 7.5 может привести к тому, что управляемые ASPX-страницы перестанут работать корректно во время выполнения дочернего запроса. Если выполняются приведенные ниже условия, дочерний запрос к `Default.aspx` документа приведет к ошибке или неожиданному поведению:

1. Страницы ASPX отправляется в браузер с **формы** элемента **действие** атрибуту присвоено значение «».
2. Форма отправляется обратно ASP.NET.
3. Управляемый HTTP-модуль считывает некоторую часть тела сущности. Например, модуль считывает **Request.Form** или **Request.Params**. Это приводит к тому, что тело сущности запроса POST считывается в управляемую память. В результате тело сущности становится недоступным для модулей машинного кода, запущенных в интегрированном режиме IIS 7 или IIS 7.5.
4. IIS **DefaultDocumentModule** объекта со временем выполняется и создает дочерний запрос к `Default.aspx` документа. Однако так как тело сущности уже было считано частью управляемого кода, тело сущности для отправки в дочерний запрос отсутствует.
5. При выполнении HTTP-конвейера для дочернего запроса обработчик `.aspx` файлов выполняется на этапе выполнения обработчика.
6. Так как тела сущности нет, нет переменных формы и ни одно состояние представления, и таким образом сведения недоступны для обработчику ASPX-страницы определить, какие события (если таковые имеются) должен вызываться. В результате для соответствующей ASPX-страницы ни один из обработчиков событий обратной передачи не выполняется.

Это поведение можно обойти следующим образом:

- Определите HTTP-модуль, который обращается к тело сущности запроса во время запросов документа по умолчанию и определить ли она может быть настроена на запуск только для запросов управляемых. В режиме интеграции с IIS 7 и IIS 7.5 модули HTTP может быть отмечен для запуска только для управляемых запросов, добавив следующий атрибут в модуль **system.webServer/modules** запись:

- `precondition="managedHandler"`

- Этот параметр отключает, запрашивает модуль для IIS 7 и IIS 7.5 определить как не управляемый запросов. Для запросов документа по умолчанию первый запрос является URL-адрес без расширений. Таким образом службы IIS не выполняется все управляемые модули, которые помечены с предусловие управляемый обработчик во время обработки исходного запроса. Таким образом управляемые модули не будут случайно прочитать тело объекта, и таким образом тела сущности по-прежнему доступна и передается для дочернего запроса и документ по умолчанию.

- Если проблемы модули HTTP должен выполняться для всех запросов (для статических файлов без расширений URL-адресов, которые разрешаются **DefaultDocumentModule** объекта, управляемого запросов, т. д.), явно изменять затронутых ASPX-страницы, Установка **действие** свойства страницы **System.Web.UI.HtmlControls.HtmlForm** управления непустой строкой. Например, если документ по умолчанию — `Default.aspx`, измените код страницы, чтобы явно задать **HtmlForm** элемента управления **действие** свойства «Default.aspx».

## <a name="changes-to-the-aspnet-code-access-security-cas-implementation"></a>Изменения в реализации разграничения доступа кода ASP.NET

ASP.NET 2.0, и по расширению использование функций ASP.NET, которые были добавлены в версии 3.5, .NET Framework 1.1 и 2.0 кода (CAS) модели управления доступом. Однако реализация управления доступом для кода в ASP.NET 4 значительно усовершенствована. В результате приложения ASP.NET с частичным доверием, которые зависят от выполнения доверенного кода в глобальный кэш сборок (GAC) может завершиться с различными исключениями безопасности. Приложения с частичным доверием, которые зависят от расширенных изменений на компьютере политику разграничения доступа кода также может произойти сбой с исключениями безопасности.

Вы можете вернуться приложения ASP.NET 4 с частичным доверием для поведение версий ASP.NET 1.1 и 2.0, с помощью нового **legacyCasModel** атрибут в **доверия** элемента конфигурации, как показано в следующем примере :

`<trust level= "Medium" legacyCasModel="true" />`

- Политика разграничения доступа кода компьютера будет учитываться.
- Допускается несколько различных уровней разрешений в одном домене приложения.
- Явное разрешение утверждения не являются обязательными для сборок в глобальном кэше СБОРОК, которые вызываются, когда только ASP.NET или другой код .NET Framework находится в стеке.

Вызов этих типов из приложений с частичным доверием не веб-больше не поддерживается в .NET Framework 4.

## <a name="membershipuser-and-other-types-in-the-systemwebsecurity-namespace-have-been-moved"></a>MembershipUser и других типов в пространстве имен System.Web.Security были перемещены

Некоторые типы, используемые в членстве ASP.NET были перемещены из `System.Web.dll` на новую сборку в сборку System.Web.ApplicationServices.dll. Типы были перемещены для устранения проблемы зависимостей уровней архитектуры между типами в клиенте и в расширенных SKU платформы .NET Framework.

Проектов веб-сайтов проблем не возникает в результате перехода этих типов, так как в сборку System.Web.ApplicationServices.dll был добавлен в список сборок, на которые имеются ссылки, используемый по умолчанию в системе компиляции ASP.NET. Если вы обновляете проект веб-сайта, который был создан с помощью более ранней версии 4 ASP.NET на ASP.NET, открыв его в Visual Studio 2010, проект будет компилироваться без ошибок.

Аналогично Если вы обновляете проект веб-приложения, который был создан в более ранней версии 4 ASP.NET на ASP.NET, открыв его в Visual Studio 2010, процесс обновления добавляет ссылку на сборку System.Web.ApplicationServices.dll в проект. Таким образом обновить Web проектов приложений также будет компилироваться без ошибок.

Скомпилированный-файлы, которые были созданы с помощью более ранних версиях ASP.NET также будут работать без ошибок в ASP.NET 4, несмотря на то, что типы членства были перемещены в другую сборку. До версии ASP.NET 4 был добавлен тип, пересылке сведений `System.Web.dll` , который автоматически направляет ссылки во время выполнения для этих типов в новое расположение для типов.

Тем не менее библиотеки классов, которые используют типы определенного членства и, были обновлены с более ранних версиях ASP.NET будет не компилироваться при использовании в проекте ASP.NET 4. ## <a name="output-caching-changes-to-vary--http-header"></a>Выходные данные кэширования изменения для изменения \* HTTP-заголовок

В ASP.NET 1.0 Ошибка приводила к тому кэшированные страницы с `Location="ServerAndClient"` как параметр кэша вывода для порождения `Vary:*` HTTP-заголовок ответа. Таким образом, клиентские браузеры получали указание никогда не кэшировать такую страницу локально.

В ASP.NET 1.1 **System.Web.HttpCachePolicy.SetOmitVaryStar** был добавлен метод, который можно вызвать для подавления `Vary:*` заголовка. Этот метод был выбран, поскольку изменение порожденную заголовка HTTP считается потенциально критические изменения во время. Тем не менее, разработчики запутывало поведением в ASP.NET, и предложить отчеты об ошибках, что разработчики не знают о существующих **SetOmitVaryStar** поведение.

В ASP.NET 4 было принято решение для устранения неполадки. `Vary:*` Заголовок HTTP больше не создается для ответов, задающих следующую директиву:

`<%@OutputCache Location="ServerAndClient" %>`

В результате **SetOmitVaryStar** больше не требуется для подавления `Vary:*` заголовка.

В приложениях, которые указывают `Location="ServerAndClient"` в **@ OutputCache** директив на странице, теперь вы увидите имя содержится в разрешении поведение **расположение** значение атрибута — это, страницы будут кэшировать в браузере без необходимости вызова **SetOmitVaryStar** метод.

Если страниц в приложении должен предоставлять `Vary:*`, вызовите **AppendHeader** метод, как показано в следующем примере:

`HttpResponse.AppendHeader("Vary","*");`

Кроме того, можно изменить значение кэширования вывода **расположение** атрибут «Server».
