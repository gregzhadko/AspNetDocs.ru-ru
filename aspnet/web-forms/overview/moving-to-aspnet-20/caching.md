# <a name="caching"></a>Caching

по [Майкрософт](https://github.com/microsoft)

> Понимание кэширования важно для хорошо выполняющегося приложения ASP.NET. ASP.NET 1. x предлагает три различных варианта кэширования. Кэширование выходных данных, кэширование фрагментов и API кэша.

Понимание кэширования важно для хорошо выполняющегося приложения ASP.NET. ASP.NET 1. x предлагает три различных варианта кэширования. Кэширование выходных данных, кэширование фрагментов и API кэша. ASP.NET 2,0 предлагает все три из этих методов, но добавляет некоторые существенные дополнительные функции. Существует несколько новых зависимостей кэша, и разработчики теперь могут также иметь возможность создавать пользовательские зависимости кэша. Настройка кэширования также значительно улучшилась в ASP.NET 2,0.

В модели на основе опроса используется триггер для таблицы базы данных, который активируется при изменении данных в таблице. ## <a name="post-cache-substitution"></a>Подстановка после кэширования

Кэширование страницы может значительно повысить производительность веб-приложения. Однако в некоторых случаях требуется, чтобы большая часть страницы была кэширована, а некоторые фрагменты страницы были динамическими. Например, при создании страницы новостей, которая является полностью статической для установки периодов времени, можно настроить кэширование всей страницы. Если вы хотите включить баннер объявления, который изменился при каждом запросе страницы, то часть страницы, содержащая объявление, должна быть динамической. Чтобы разрешить кэширование страницы, но заменять некоторое содержимое динамически, можно использовать подстановку ASP.NET после кэширования. При подстановке после кэширования вся страница кэшируется на выходе с конкретными частями, помеченными как исключенные из кэширования. В примере баннеров баннера элемент управления AdRotator позволяет воспользоваться преимуществами подстановки после кэширования, чтобы баннеры динамически создавались для каждого пользователя и для каждого обновления страницы.

Существует три способа реализации подстановки после кэширования:

- Декларативно, с помощью элемента управления Substitution.
- Программно с помощью API подстановки.
- Неявным образом с помощью элемента управления AdRotator.

### <a name="substitution-control"></a>Элемент управления Substitution

Элемент управления Substitution ASP.NET указывает раздел кэшированной страницы, который создается динамически, а не кэшируется. Элемент управления Substitution помещается в расположение на странице, где должно отображаться динамическое содержимое. Во время выполнения элемент управления Substitution вызывает метод, указанный с помощью свойства имя_метода. Метод должен возвращать строку, которая затем заменяет содержимое элемента управления Substitution. Метод должен быть статическим методом на содержащей странице или элементе управления UserControl. Использование элемента управления Substitution приводит к изменению кэша на стороне клиента на возможность кэширования сервера, поэтому страница не кэшируется на клиенте. Это гарантирует, что будущие запросы к странице снова вызывают метод для создания динамического содержимого.

### <a name="substitution-api"></a>API подстановки

Чтобы создать динамическое содержимое для кэшированной страницы программным способом, можно вызвать метод [вритесубститутион](https://msdn.microsoft.com/library/system.web.httpresponse.writesubstitution.aspx) в коде страницы, передав ему имя метода в качестве параметра. Метод, обрабатывающий создание динамического содержимого, принимает один параметр [HttpContext](https://msdn.microsoft.com/library/system.web.httpcontext.aspx) и возвращает строку. Возвращаемая строка — это содержимое, которое будет заменено в указанном расположении. Преимущество вызова метода Вритесубститутион вместо использования элемента управления Substitution декларативно заключается в том, что можно вызвать метод любого произвольного объекта, а не вызывать статический метод страницы или объекта UserControl.

Вызов метода Вритесубститутион приводит к изменению кэша на стороне клиента на возможность кэширования сервера, поэтому страница не кэшируется на клиенте. Это гарантирует, что будущие запросы к странице снова вызывают метод для создания динамического содержимого.

### <a name="adrotator-control"></a>Элемент управления AdRotator

Серверный элемент управления AdRotator реализует поддержку подстановки после кэширования внутренним образом. Если поместить элемент управления AdRotator на страницу, он будет отображать уникальные объявления для каждого запроса, независимо от того, кэшируется ли родительская страница. В результате страница, включающая элемент управления AdRotator, кэшируется только на стороне сервера.

## <a name="controlcachepolicy-class"></a>Класс Контролкачеполици

Класс Контролкачеполици позволяет программным образом управлять кэшированием фрагментов с помощью пользовательских элементов управления. ASP.NET внедряет пользовательские элементы управления в экземпляре [басепартиалкачингконтрол](https://msdn.microsoft.com/library/system.web.ui.basepartialcachingcontrol.aspx) . Класс Басепартиалкачингконтрол представляет пользовательский элемент управления, для которого включено кэширование вывода.

При доступе к свойству [басепартиалкачингконтрол. CachePolicy](https://msdn.microsoft.com/library/system.web.ui.basepartialcachingcontrol.cachepolicy.aspx) элемента управления [партиалкачингконтрол](https://msdn.microsoft.com/library/system.web.ui.partialcachingcontrol.aspx) всегда будет получен допустимый объект контролкачеполици. Однако при доступе к свойству [UserControl. CachePolicy](https://msdn.microsoft.com/library/system.web.ui.usercontrol.cachepolicy.aspx) элемента управления [UserControl](https://msdn.microsoft.com/library/system.web.ui.usercontrol.aspx) вы получаете допустимый объект контролкачеполици, только если пользовательский элемент управления уже заключен в оболочку элемента управления басепартиалкачингконтрол. Если он не заключен в оболочку, то объект Контролкачеполици, возвращаемый свойством, вызовет исключения при попытке управлять им, так как у него нет связанного Басепартиалкачингконтрол. Чтобы определить, поддерживает ли экземпляр UserControl кэширование без создания исключений, проверьте свойство [суппортскачинг](https://msdn.microsoft.com/library/system.web.ui.controlcachepolicy.supportscaching.aspx) .

Использование класса Контролкачеполици является одним из нескольких способов включения кэширования вывода. В следующем списке описываются методы, которые можно использовать для включения кэширования вывода.

- Используйте директиву [@ OutputCache](https://msdn.microsoft.com/library/hdxfb6cy.aspx) , чтобы включить кэширование вывода в декларативных сценариях.
- Используйте атрибут [партиалкачингаттрибуте](https://msdn.microsoft.com/library/system.web.ui.partialcachingattribute.aspx) , чтобы включить кэширование для пользовательского элемента управления в файле кода программной части.
- Используйте класс Контролкачеполици, чтобы указать параметры кэша в программных сценариях, в которых вы работаете с экземплярами Басепартиалкачингконтрол, которые были включены в кэш с помощью одного из предыдущих методов и динамически загружаются с помощью метода [System. Web. UI. темплатеконтрол. LoadControl](https://msdn.microsoft.com/library/system.web.ui.templatecontrol.loadcontrol.aspx) .

Экземпляр Контролкачеполици может быть успешно обработан только между этапами init и PreRender жизненного цикла элемента управления. Если изменить объект Контролкачеполици после этапа PreRender, ASP.NET создает исключение, так как любые изменения, внесенные после отрисовки элемента управления, фактически не влияют на параметры кэша (элемент управления кэшируется на стадии подготовки к просмотру). Наконец, экземпляр пользовательского элемента управления (и, следовательно, его объект Контролкачеполици) доступен только для программной манипуляции, когда он фактически отображается.

### <a name="more-information"></a>Дополнительные сведения

Есть некоторые дополнительные сведения, о которых следует знать при настройке кэша.

- Если ограничение байтов исключительного пользования рабочего процесса не задано, кэш будет использовать один из следующих ограничений: 

    - x86 ГБ: 800MB или 60% физической памяти, в зависимости от того, что меньше
    - x86 3 ГБ: 1800MB или 60% физической памяти, в зависимости от того, что меньше
    - x64:1 ТБ или 60% физической памяти, в зависимости от того, что меньше
- Если задано ограничение байтов исключительного пользования для рабочего процесса и &lt;кэша Приватебитеслимит/&gt;, то кэш будет использовать минимум из двух.
- Как и в 1. x, мы удаляем записи кэша и вызываеми GC. Собирайте по двум причинам: 

    - Мы очень близки к ограничению байтов исключительного объема
    - Объем доступной памяти приближается к 10%
- Можно эффективно отключить усечение и кэширование для нехватки памяти, установив для &lt;кэша Перцентажефисикалмеморюселимит/&gt; значение 100.
- В отличие от 1. x, 2,0 будет приостанавливать вызовы Trim и сбора, если последний GC. При этом не уменьшается число байтов исключительного хранения или размер управляемых куч превышает 1% от предела памяти (кэша).
