## <a name="introduction"></a>Вступление

ASP.NET версии 2.0 включает ряд *служб приложений*, которой являются частью платформы .NET Framework и выступающие в качестве служб набор стандартных блоков, которые можно использовать для добавления широкий набор функций для веб-приложения. К службам приложений относятся:

- **Членство** — API для создания и управления учетными записями пользователей.
- **Роли** — API для упорядочивания пользователей по группам.
- **Профиль** — API для хранения пользовательского содержимого конкретного пользователя.
- **Карта сайта** — API для определения сайта s логическая структура в виде иерархии, который затем может отображаться с помощью элементов управления навигации, например меню и строки навигации.
- **Персонализация** — API для поддержки настройки, наиболее часто используется с [ *WebParts*](https://msdn.microsoft.com/library/e0s9t4ck.aspx).
- **Наблюдение за работоспособностью** — API для мониторинга производительности, безопасности, ошибки и другие метрики работоспособности системы для запущенного веб-приложения.
  

Интерфейсы API службы приложений не привязаны к конкретной реализации. Вместо этого вы указываете службы приложений, чтобы использовать конкретную *поставщика*, и этот поставщик реализует службу с помощью определенной технологии. Наиболее часто используемые поставщики для веб-приложений, размещенных на веб-хостинга являются тех поставщиков, которые используется реализация базы данных SQL Server. Например `SqlMembershipProvider` является поставщиком для API членства, который хранит данные учетной записи пользователя в базе данных Microsoft SQL Server.

С помощью службы приложений и поставщики SQL Server добавляет некоторые проблемы, при развертывании приложения. Во-первых объекты базы данных служб приложения должен быть надлежащим образом создан для разработки и рабочей баз данных и соответствующим образом инициализирован. Кроме того, существуют также важных параметров конфигурации, которые необходимо внести.

> [!NOTE]
> Службы приложений, интерфейсы API были разработаны с помощью [ *модель поставщика*](http://aspnet.4guysfromrolla.com/articles/101905-1.aspx), шаблон, который позволяет API s детали реализации должны быть предоставлены во время выполнения. .NET Framework поставляется с рядом поставщиков услуг, которые могут использоваться, например `SqlMembershipProvider` и `SqlRoleProvider`, которые являются поставщиками членства и ролей API-интерфейсов, использующих SQL Server, базы данных реализации. Вы также можете создать и подключаемый модуль пользовательского поставщика. На самом деле рецензий веб-приложения уже содержит настраиваемый поставщик API Map сайта (`ReviewSiteMapProvider`), который создает карту узла из данных в `Genres` и `Books` таблицы в базе данных.


Этот учебник начинается со взгляда на том, как я дополнил рецензий веб-приложения для использования членства и ролей API-интерфейсы. ## <a name="deploying-a-web-application-that-uses-application-services"></a>Развертывание веб-приложения, использующего службы приложений

При развертывании веб-сайта, использующего службы приложений и поставщика, который хранит сведения о службах приложений в базе данных, крайне важно, что объекты базы данных, необходимые для приложения служб быть создана в рабочей базе данных. Изначально рабочей базы данных не содержит этими объектами, поэтому при первом развертывании приложения (или при его развертывании в первый раз после добавления службы приложений), вы должны предпринять дополнительные действия, чтобы получить эти объекты необходимые базы данных на рабочей базы данных.

Еще одна проблема может возникнуть при развертывании веб-сайта, использующего службы приложений, если вы намереваетесь реплицировать учетные записи пользователей, созданные в среде разработки в рабочую среду. В зависимости от конфигурации членства и ролей вполне возможно, что даже если вы успешно скопировать учетные записи пользователей, которые были созданы в среде разработки в производственной базе данных, эти пользователи не могут войти в веб-приложение в рабочей среде. Мы рассмотрим причины этой проблемы и обсуждаются способы избежать.

В ASP.NET имеется прекрасное [ *средство администрирования веб-сайта (WSAT)* ](https://msdn.microsoft.com/library/yy40ytx0.aspx) , можно запустить из Visual Studio и позволяет пользователю учетной записи, роли и авторизация правил, которые управляются с помощью веб- интерфейс. К сожалению WSAT работает только для локальных веб-сайтов, это означает, что он не может использоваться для удаленного управления учетных записей пользователей, роли и правила авторизации для веб-приложения в рабочей среде. Мы рассмотрим различные способы реализации поведения, схожего WSAT с веб-сайта в рабочей среде.

Другой вариант — `aspnet_regsql.exe` средство, которое добавляет или удаляет объекты базы данных приложения службы из базы данных.

Он не выполняет миграцию данных в этих объектах базы данных из базы данных разработки в производственной базе данных. Если требуется скопировать данные учетной записи и роль пользователя в базе данных разработки в производственной базе данных, использовать технологии, описанные в *развертывание базы данных* руководства.

### <a name="copying-user-accounts-from-development-to-production"></a>Копирование учетных записей пользователей из среды разработки в рабочей среде

При использовании `SqlMembershipProvider` и `SqlRoleProvider` классы поставщиков для хранения сведений о приложении служб в базе данных SQL Server, учетную запись и роль сведения о пользователе хранятся в различных таблицах базы данных, включая `aspnet_Users`, `aspnet_Membership`, `aspnet_Roles`, и `aspnet_UsersInRoles`, среди прочего. При создании учетных записей пользователей во время разработки в среде разработки можно реплицировать эти учетные записи пользователей в рабочей среде путем копирования соответствующих записей из таблиц соответствующей базой данных. Если вы использовали мастер публикации баз данных для развертывания объектов базы данных служб приложения, может также выбрано копирование записей, что приведет к учетные записи пользователей, созданные в разработке также размещались в рабочей среде. Но в зависимости от параметров конфигурации, может оказаться, что этих пользователей, учетные записи которых были созданы в разработке и копируются в рабочую среду не удается выполнить вход на веб-сайте в рабочей среде. Что же делать?

`SqlMembershipProvider` И `SqlRoleProvider` классы поставщика были разработаны таким образом, что отдельной базы данных можно использовать в качестве пользовательского хранилища для нескольких приложений, где каждое приложение может теоретически иметь пользователей с помощью перекрывающиеся имен пользователей и ролей с тем же именем. Для обеспечения этой гибкости, базы данных ведет список приложений в `aspnet_Applications` таблицы и каждый пользователь связан с одним из этих приложений. В частности `aspnet_Users` таблица имеет `ApplicationId` столбец, который связывает каждому пользователю осуществлять запись в `aspnet_Applications` таблицы.

В дополнение к `ApplicationId` столбце `aspnet_Applications` таблица также содержит `ApplicationName` столбец, который предоставляет имя более понятную для приложения. При попытке веб-сайт для работы с учетной записи пользователя, такие как проверка учетных данных пользователя s со страницы входа, его необходимо сообщить `SqlMembershipProvider` класса какое приложение для работы с. Обычно делает это, указав имя приложения и это значение поступает из конфигурации поставщика s в `Web.config` — в частности, с помощью `applicationName` атрибута.

Но что произойдет, если `applicationName` атрибут не указан в `Web.config`? В этом случае членство система использует в корневом каталоге приложения, как `applicationName` значение. Если `applicationName` атрибут не задан явно `Web.config`, затем, есть вероятность, что среда разработки и рабочей среде использовать корневой другое приложение, которое таким образом будет связан с другого приложения имена в службах приложений. В случае подобные различия этих пользователей, созданных в среде разработки будет иметь `ApplicationId` значение, которое не совпадает с `ApplicationId` значение для рабочей среды. Конечным результатом является то, что эти пользователи не смогут выполнить вход.

> [!NOTE]
> Если вы окажетесь в такой ситуации — с учетными записями пользователей, которые копируются в рабочую среду с несоответствующее `ApplicationId` значения — можно написать запрос, чтобы обновить эти неверные `ApplicationId` значения `ApplicationId` использовать в рабочей среде. После обновления, пользователи, учетные записи которых были созданы с помощью среды разработки теперь будет возможность входа в веб-приложение в рабочей среде.


Хорошая новость — это простое действие, можно предпринять для убедитесь, что две среды используйте тот же `ApplicationId` — явным образом переводит `applicationName` атрибут в `Web.config` для всех поставщиков служб вашего приложения. Явно задать `applicationName` атрибут «BookReviews» в `<membership>` и `<roleManager>` элементы, что этот фрагмент кода из `Web.config` показано.
