по [том Дайкстра](https://github.com/tdykstra)

> В этом разделе описываются параметры для создания веб-проектов ASP.NET в Visual Studio 2013 с обновлением 3
> 
> Ниже приведены некоторые из новых функций для веб-разработки, по сравнению с более ранних версиях Visual Studio:
> 
> - Простой пользовательский Интерфейс для создания проектов предложения [поддержка нескольких платформ ASP.NET](#add) (веб-форм, MVC и веб-API).

> Сведения о способах создания веб-проектов для облачных служб Azure или мобильные службы Azure, см. в разделе [начало работы с облачными службами Azure и ASP.NET](https://azure.microsoft.com/documentation/articles/cloud-services-dotnet-get-started/) и [Создание приложения списка лидеров с помощью .NET мобильных служб Azure Серверной части](https://azure.microsoft.com/documentation/articles/mobile-services-dotnet-backend-windows-store-dotnet-leaderboard/).

## <a name="overview-of-web-application-project-creation"></a>Обзор создания проекта веб-приложений

Ниже показано, как создать веб-проекта:

1. Нажмите кнопку **новый проект** в **запустить** страницы или в **файл** меню.
2. В **новый проект** диалоговое окно, нажмите кнопку **Web** в левой области и **веб-приложение ASP.NET** в средней области.

    ![Диалоговое окно создания нового проекта](creating-web-projects-in-visual-studio/_static/image1.png)

    Вы можете выбрать **Cloud** на левой панели, чтобы создать [облачной службы Azure](https://docs.microsoft.com/azure/cloud-services/cloud-services-how-to-create-deploy), [мобильной службы Azure](https://msdn.microsoft.com/library/windows/apps/dn629482.aspx), или [веб-задания Azure](https://azure.microsoft.com/documentation/articles/web-sites-dotnet-deploy-webjobs). В этом разделе не рассматривается эти шаблоны.
3. В области справа щелкните **добавить Application Insights в проект** флажок, если требуется работоспособность и мониторинг использования приложения. Дополнительные сведения см. в разделе [наблюдения за производительностью в веб-приложений](https://azure.microsoft.com/documentation/articles/app-insights-web-monitor-performance/).
4. Укажите проект **имя**, **расположение**и другие параметры, а затем щелкните **ОК**.

    **Новый проект ASP.NET** откроется диалоговое окно.

    ![Диалоговое окно создания нового проекта](creating-web-projects-in-visual-studio/_static/image2.png)
5. Выберите шаблон.

    ![Выберите шаблон](creating-web-projects-in-visual-studio/_static/image3.png)
6. Если вы хотите добавить поддержку дополнительных платформ, не включенный в шаблон, щелкните соответствующий флажок. (В приведенном примере, можно добавить MVC или веб-API в проект веб-форм.)

    ![Добавьте платформы](creating-web-projects-in-visual-studio/_static/image4.png)
7. <a id="testproj"></a>Если вы хотите добавить проект модульного теста, нажмите кнопку **Добавление модульных тестов**.

    ![Добавление модульных тестов](creating-web-projects-in-visual-studio/_static/image5.png)
8. Если требуется другой метод проверки подлинности чем шаблон по умолчанию, нажмите кнопку **изменить способ проверки подлинности**.

    ![Настройка проверки подлинности кнопки](creating-web-projects-in-visual-studio/_static/image6.png)

    ![Настройка диалогового окна проверки подлинности](creating-web-projects-in-visual-studio/_static/image7.png)

<a id="empty"></a>
### <a name="empty-template"></a>Пустой шаблон

Пустой шаблон предоставляет только минимальный папки и файлы для веб-приложение ASP.NET, такие как файл проекта (*.csproj* или. *VBPROJ*) и *Web.config* файл. Можно добавить поддержку для веб-форм, MVC или веб-API с помощью флажков в списке **добавить папки и основные ссылки для:** метки.

Параметры проверки подлинности не доступны для пустого шаблона. Функциональные возможности проверки подлинности реализуется в примерах приложений, и пустой шаблон не создает пример приложения.

<a id="wf"></a>
### <a name="web-forms-template"></a>Веб-форм шаблона

Веб-форм, framework предоставляет следующие возможности, которые позволяют быстро создавать веб-сайты, обладающих богатыми возможностями пользовательского интерфейса и данных, доступ к функциям:

- Конструктора WYSIWYG в Visual Studio.
- Серверные элементы управления, которые отображают HTML и, которые можно настраивать путем настройки свойств и стили.
- Широкий набор элементов управления, для доступа к данным и отображение данных.
- Модель событий, которая предоставляет события, которые вы можете программировать, как необходимо запрограммировать клиентского приложения, таких как WPF.
- Автоматическое сохранение состояния (данных) между запросами HTTP.

Как правило Создание приложения Web Forms требует меньше усилий программирования, чем создание того же приложения с помощью платформы ASP.NET MVC. Тем не менее веб-форм не только для быстрой разработки приложений. Существует множество сложных коммерческих приложений и платформ, созданная на основе веб-форм.

Так как страницы веб-форм и элементов управления на странице автоматически создают большую часть разметки, который отправляется в браузер, у вас нет вида HTML, ASP.NET MVC предоставляет возможность точного управления. Декларативная модель для настройки страниц и элементов управления позволяет свести к минимуму объем кода, нужно писать, но скрывает некоторые поведения HTML и HTTP. Например не всегда можно указать точно разметку HTML могут быть созданы элементом управления.

Веб-форм недоступно так же легко, как ASP.NET MVC для методов разработки, основанный на шаблонах например [управляемой тестами разработки](http://en.wikipedia.org/wiki/Test-driven_development), [Разделение областей ответственности](http://en.wikipedia.org/wiki/Separation_of_concerns), [инверсии элемент управления](http://en.wikipedia.org/wiki/Inversion_of_control), и [внедрения зависимостей](http://en.wikipedia.org/wiki/Dependency_injection). Если вы хотите написать код добавляются таким образом, вы можете; Это не просто как автоматический, как в платформе ASP.NET MVC. [ASP.NET Web Forms MVP](http://webformsmvp.com/) проекта показан подход, который облегчает разделения проблем и пригодности для тестирования при сохранении быстрой разработки веб-форм, созданная для доставки. Microsoft SharePoint построена на Web Forms MVP.

Шаблон веб-форм создает пример приложения Web Forms, использующего [Bootstrap](#bootstrap) быстрые функции проектирования и темы. На следующем рисунке на домашнюю страницу.

<a id="mvc"></a>
### <a name="mvc-template"></a>Шаблон MVC

ASP.NET MVC была разработана для упрощения методики разработки, основанный на шаблонах, таких как [управляемой тестами разработки](http://en.wikipedia.org/wiki/Test-driven_development), [Разделение областей ответственности](http://en.wikipedia.org/wiki/Separation_of_concerns), [инверсии управления](http://en.wikipedia.org/wiki/Inversion_of_control), и [внедрения зависимостей](http://en.wikipedia.org/wiki/Dependency_injection). Эта платформа стимулирует разделение уровня бизнес-логики, веб-приложения из его уровень представления данных. Путем разделения приложения на моделях (M), представления (V) и контроллерах (C), ASP.NET MVC можно упростить управление сложностью в более крупных приложений.

ASP.NET MVC вы более напрямую работать с HTML и протоколу HTTP, чем веб-форм. Например веб-форм можно автоматически сохранять состояние, которое между HTTP-запросы, но у вас есть код, который явно в MVC. Модели MVC удобен тем, что он позволяет получить полный контроль над именно то, что делает ваше приложение и его поведение в среде веб. Недостатком является то, что вам нужно писать дополнительный код.

Шаблон MVC обеспечивает возможность расширения, предоставляя разработчикам power возможность настройки платформы для удовлетворяют требованиям приложения. Кроме того исходный код ASP.NET MVC доступна на условиях OSI лицензии.

Шаблон создает образец приложения MVC 5, который использует [Bootstrap](#bootstrap) быстрые функции проектирования и темы. На следующем рисунке на домашнюю страницу.

## <a name="adding-support-for-additional-frameworks"></a>Добавить поддержку дополнительных платформ

При выборе шаблона автоматически установлен флажок для платформы, используемых шаблоном. Например, если вы выберите **веб-форм** шаблона, **веб-форм** флажок установлен и снять его нельзя.

![Выберите шаблон](creating-web-projects-in-visual-studio/_static/image21.png)

![Добавьте платформы](creating-web-projects-in-visual-studio/_static/image22.png)

Можно установить флажок для платформы, не включенные в шаблон, чтобы добавить поддержку для этой платформы, при создании проекта. Например, чтобы включить использование веб-форм *.aspx* страниц, если бы вы выбрали шаблон, выберите **веб-форм** "флажок". Или чтобы MVC при использовании шаблона веб-форм, нажмите кнопку **MVC** "флажок". Добавление framework включает поддержку времени разработки, а также во время выполнения. Например при добавлении поддержки MVC в проект веб-форм, можно для формирования шаблонов контроллеров и представлений.

Если объединить Web Forms и MVC в проекте и включите [Friendly URLs](http://www.hanselman.com/blog/IntroducingASPNETFriendlyUrlsCleanerURLsEasierRoutingAndMobileViewsForASPNETWebForms.aspx) в Web Forms, может существовать непредвиденных неполадок маршрутизации, где один URL-адрес имеет несколько возможных целевых объектов. Маршруты, которые определены сначала будет иметь приоритет. Например, если у вас есть `Home` контроллера и *Home.aspx* странице `http://contoso.com/home` URL-адрес будет отправлена *Home.aspx* при вызове метода `EnableFriendlyUrls` метод перед вызовом метода `MapRoute`метод в *RouteConfig.cs*, или же URL-адрес будет перейти к представлению по умолчанию для вашей `Home` контроллера при вызове метода `MapRoute` перед `EnableFriendlyUrls`.

Добавление framework не обеспечивает функциональные возможности примера приложения. Например, при добавлении веб-формы поддерживают выбрав шаблон, не *Default.aspx* создается файл домашнюю страницу. Добавляются только папки, файлы и ссылки, необходимые для поддержки платформы. По этой причине при добавлении платформы не изменяется параметры проверки подлинности, которые реализованы в коде в примерах приложений, созданных с использованием шаблонов. Например, при выборе пустого шаблона и добавить веб-форм или MVC поддерживает, **Настройка проверки подлинности** кнопка будет по-прежнему отключена.

В следующих разделах кратко последствия каждого флажка.

### <a name="add-web-forms-support"></a>Добавление поддержки Web Forms

Создает пустой *приложения\_данных* и *моделей* папок и *Global.asax* файл. Они уже созданы, все шаблоны, кроме пустой шаблон, установив флажок веб-форм нет разницы для других шаблонов.

Шаблон веб-форм включает понятное URL-адреса, по умолчанию, но при добавлении веб-формы поддерживают на другие шаблоны, установив флажок веб-форм, которые удобных URL-адресов не включается автоматически.

### <a name="add-mvc-support"></a>Добавить поддержку MVC

Устанавливает пакеты MVC Razor и NuGet веб-страницы, создает пустой *приложения\_данных*, *контроллеров*, *моделей*, и *представления*папок, создает *приложения\_запустить* папка с *RouteConfig.cs* файл и создает *Global.asax* файл.

### <a name="add-web-api-support"></a>Добавление поддержки Web API

Устанавливает пакеты веб-API и Newtonsoft.Json NuGet, создает пустой *приложения\_данных*, *контроллеров*, и *моделей* папок, создает  *Приложение\_запустить* папка с *WebApiConfig.cs* файл и создает *Global.asax* файл.

## <a name="authentication-methods"></a>Методы проверки подлинности

Visual Studio 2013 предлагает несколько вариантов проверки подлинности для веб-форм, MVC и веб-API шаблонов:

- [Без проверки подлинности](#noauth)
- [Учетные записи отдельных пользователей](#indauth) (удостоверения ASP.NET, которая ранее называлась членства ASP.NET)
- [Учетные записи организации](#orgauth) (Windows Server Active Directory или Azure Active Directory)
- [Проверка подлинности Windows](#winauth) (интрасети)

![Настройка диалогового окна проверки подлинности](creating-web-projects-in-visual-studio/_static/image23.png)

<a id="noauth"></a>

### <a name="no-authentication"></a>Без проверки подлинности

При выборе **без проверки подлинности**, пример приложения будет содержать нет веб-страницы для входа в систему, не пользовательского интерфейса, указывающий, который входит в систему, не классы сущностей для базы данных членства, а не строку подключения для базы данных членства.

<a id="indauth"></a>
### <a name="individual-user-accounts"></a>Учетные записи отдельных пользователей

При выборе **учетные записи отдельных пользователей**, пример приложения будет настроен для использования ASP.NET Identity (прежнее название членства ASP.NET) для проверки подлинности пользователя. ASP.NET Identity позволяет зарегистрировать учетную запись, можно создать имя пользователя и пароль на сайте или вход с помощью поставщиков социальных сетей, таких как Facebook, Google, учетная запись Microsoft или Twitter. Хранилища данных по умолчанию для профилей пользователей в ASP.NET Identity является базой данных SQL Server LocalDB, который развертывается в SQL Server или базы данных SQL Azure для рабочего сайта.

В Visual Studio 2013 эти функции отличаются от как в Visual Studio 2012, но был переписан в исходный код для системы членства ASP.NET. Новый базовый код следующие преимущества:

- Новая система членства основан на [OWIN](http://owin.org/) вместо того чтобы модуль проверки подлинности форм ASP.NET. Это означает, что можно использовать один и тот же механизм проверки подлинности ли вы используете веб-форм или MVC в службах IIS, или самостоятельно вы размещаете, веб-API или SignalR.
- Новая база данных членства управляется Entity Framework Code First, и все таблицы представлены классы сущностей, которые можно изменить. Это означает, что вы можете легко настроить схему базы данных и связанных с профилем пользовательского веб-интерфейса в соответствии с собственными требованиями, и вы можете легко развернуть обновления с помощью Code First Migrations.

Новая система членства реализуется автоматически в новых шаблонах, и он может быть реализован вручную в любой проект, предназначенного для .NET 4.5 или более поздней версии.

ASP.NET Identity является хорошим выбором при создании веб-узлы IIS, который является главным образом для внешних клиентов. Если ваша организация использует Active Directory или Office 365 и хотите, чтобы создать проект, который позволяет single-sign-on для сотрудникам и деловым партнерам, **учетные записи организации** вариантом может быть лучшим выбором.

Дополнительные сведения о параметре учетные записи отдельных пользователей см. следующие ресурсы:

- [www.ASP.NET/Identity](../../../identity/index.md). Документацию по ASP.NET Identity на веб-сайте ASP.NET.
- [Создать приложение ASP.NET MVC 5 с помощью Facebook и Google OAuth2 и OpenID единого входа](../../../mvc/overview/security/create-an-aspnet-mvc-5-app-with-facebook-and-google-oauth2-and-openid-sign-on.md). Также показано, как настроить данные профиля пользователя.
- [Веб-API — внешние службы проверки подлинности](../../../web-api/overview/security/external-authentication-services.md)
- [Добавление внешних имен входа в приложение ASP.NET в Visual Studio 2013](https://blogs.msdn.com/b/webdev/archive/2013/06/27/adding-external-logins-to-your-asp-net-application-in-visual-studio-2013.aspx)

<a id="orgauth"></a>
### <a name="organizational-accounts"></a>Учетные записи организации

При выборе **учетные записи организации**, будет настроен пример приложения Windows Identity Foundation (WIF) для проверки подлинности на основе учетных записей пользователей в Azure Active Directory (Azure AD, который включает в себя Office 365) или Windows Server Active Directory. Дополнительные сведения см. в разделе [параметры проверки подлинности учетной записи организации](#orgauthoptions) далее в этом разделе.

<a id="winauth"></a>
### <a name="windows-authentication"></a>Проверка подлинности Windows

При выборе **проверки подлинности Windows**, пример приложения будет настроен для использования модуля IIS проверку подлинности Windows для проверки подлинности. Идентификатор домена и пользователя Active directory или учетную запись локального компьютера, который выполнил вход в Windows, но не включают регистрацию пользователя или журнала в пользовательском Интерфейсе будет отображаться в приложении. Этот параметр предназначен для веб-сайтов интрасети.

Кроме того, можно создать сайт интрасети, использующее проверку подлинности AD, выбрав [локального параметра в разделе учетных записей организации](#orgauthonprem). Параметр On-Premises использует Windows Identity Foundation (WIF) вместо модуль проверки подлинности Windows. Чтобы настроить параметр On-Premises необходимы некоторые дополнительные действия, но WIF включает возможности, недоступные в модуле проверки подлинности Windows. Например с применением WIF можно настроить доступ к приложениям в Active Directory и запроса данных каталога.

<a id="orgauthoptions"></a>
## <a name="organizational-account-authentication-options"></a>Параметры проверки подлинности учетной записи организации

**Настройка проверки подлинности** диалоговое окно предлагает несколько вариантов для Azure Active Directory (Azure AD, который включает в себя Office 365) или проверки подлинности учетной записи Windows Server Active Directory (AD):

- [Облако — одна организация](#orgauthsingle) (Azure AD или AD, используя Интеграция каталогов с Azure AD)
- [Облако — организации с несколькими](#orgauthmulti) (Azure AD или AD, используя Интеграция каталогов с Azure AD)

Если попробовать один из параметров Azure AD, но у учетной записи еще, [щелкните здесь, чтобы зарегистрировать учетную запись Azure AD](https://go.microsoft.com/fwlink/?LinkId=309942).

> [!NOTE]

Введите имя и пароль для учетной записи организации (например, admin@contoso.onmicrosoft.com), имеет права администратора для вашего клиента Azure AD.
> 
> **Не вводите учетные данные для учетной записи Майкрософт (например, contoso@hotmail.com) в поле диалогового окна входа.**

<a id="orgauthsingle"></a>
### <a name="cloud---single-organization-authentication"></a>Облако — проверка подлинности одной организации

![Проверка подлинности одной организации](creating-web-projects-in-visual-studio/_static/image24.png)

Выберите этот параметр, если вы хотите включить проверку подлинности для учетных записей пользователей, которые определены в Azure AD в один [клиента](https://technet.microsoft.com/library/jj573650.aspx). Например сайт — contoso.com, и он будет предоставляться сотрудников компании Contoso, которые находятся в клиенте contoso.onmicrosoft.com. Нельзя настроить Azure AD, чтобы пользователи из других клиентов, доступ к приложению.

#### <a name="domain"></a>Домен

Введите имя домена Azure AD, который вы хотите настроить приложение в, например: `contoso.onmicrosoft.com`. Если у вас есть [личного домена](http://www.cloudidentity.com/blog/2013/04/14/adding-a-custom-domain-to-your-windows-azure-ad/), такие как `contoso.com` вместо `contoso.onmicrosoft.com`, вы можете указать его здесь.

#### <a name="access-level"></a>Уровень доступа

Если приложение должно выполнить запрос или обновить сведения о каталоге с помощью Graph API, выберите **единый вход, чтение данных каталога** или **единый вход, чтение и запись данных каталога**. В противном случае выберите **единого входа**. Дополнительные сведения см. в разделе [уровни доступа приложения](https://msdn.microsoft.com/library/windowsazure/b08d91fa-6a64-4deb-92f4-f5857add9ed8#BKMK_AccessLevels) и [с помощью API Graph для запроса Azure AD](https://msdn.microsoft.com/library/windowsazure/dn151791.aspx).

#### <a name="application-id-uri"></a>URI идентификатора приложения

По умолчанию шаблон создает URI идентификатора приложения можно путем добавления имени проекта в домен Azure AD. Например, если имя проекта — `Example` и используется домен `contoso.onmicrosoft.com`, становится URI идентификатора приложения `https://contoso.onmicrosoft.com/Example`. Если вы хотите вручную указать URI идентификатора приложения, разверните **Дополнительно** разделе и в текстовом поле введите URI идентификатора приложения. Приложения, идентификатор URI должен начинаться с `https://`.

По умолчанию если приложение, который уже был подготовлен в Azure AD имеет же URI идентификатора приложения, что Visual Studio использует для проекта, проект будет подключен к существующему приложению вместо его подготовки. Если вы хотите нового приложения должны быть подготовлены в этом случае, снимите **перезаписать запись приложения в том случае, если с таким же Идентификатором уже существует** "флажок".

Если **перезаписать** флажок снят и Visual Studio находит существующее приложение с тем же URI идентификатора приложения, он создает новый универсальный код Ресурса, добавляя номер к URL-АДРЕСУ, его нужно было использовать. Например, предположим, что имя проекта — `Example`, оставьте поле пустым, необходимо очистить **перезаписать** "флажок", а в клиенте Azure AD уже есть приложение с URI `https://contoso.onmicrosoft.com/Example`. В этом случае новое приложение будет использован как URI идентификатора приложения `https://contoso.onmicrosoft.com/Example_20130619330903`.

#### <a name="provisioning-the-application-in-azure-ad"></a>Подготовка приложения в Azure AD

Чтобы подготовить приложение в Azure AD или подключение проекта к существующему приложению, Visual Studio требуются учетные данные глобального администратора для домена. При нажатии кнопки **ОК** в **Настройка проверки подлинности** диалоговом окне появится имя пользователя и пароль глобального администратора для домена, вы указали. Позже, при нажатии кнопки **Создание проекта** в **новый проект ASP.NET** диалоговом окне Visual Studio выполняет подготовку приложения в Azure AD. Обратите внимание на то, что в ходе этого процесса Visual Studio внедряет значения секрета клиента в файле Web.config, который срок действия один год после ее создания.

Сведения о создании приложений, использующих **облако — одна организация** проверки подлинности, ознакомьтесь со следующими ресурсами:

### <a name="cloud---multi-organization-authentication"></a>Облако — проверка подлинности организации с несколькими

![Несколько аутентификацию организации](creating-web-projects-in-visual-studio/_static/image25.png)

<a id="orgauthonprem"></a>
### <a name="on-premises-organizational-authentication"></a>Проверка подлинности в локальной организации

![Проверка подлинности в локальной организации](creating-web-projects-in-visual-studio/_static/image26.png)

Выберите этот параметр, если вы хотите включить проверку подлинности для учетных записей пользователей, которые определены в Windows Server Active Directory (AD), и вы не хотите использовать Azure AD. Этот параметр можно использовать для создания сайта интрасети или веб-узла. Для веб-сайтов используйте службы федерации Active Directory (ADFS) для предоставления доступа к AD. Дополнительные сведения см. в разделе [использовать параметр организации проверки подлинности с локальным (ADFS) с помощью ASP.NET в Visual Studio 2013](http://www.cloudidentity.com/blog/2014/02/12/use-the-on-premises-organizational-authentication-option-adfs-with-asp-net-in-visual-studio-2013/).

Для сайта интрасети, в качестве альтернативы можно выбрать [проверки подлинности Windows](#winauth) вместо этого параметр. Для проверки подлинности Windows не нужно предоставить URL-адрес документа метаданных. Тем не менее проверка подлинности Windows не предоставляют следующие возможности для управления доступом к приложениям в Active Directory или запросить данные каталога.

#### <a name="on-premises-authority"></a>Локальный центр

Введите URL-адрес, указывающий на документ метаданных. Документ метаданных содержит координаты центра. Приложение будет использовать их для потока web единого входа.

#### <a name="application-id-uri"></a>URI идентификатора приложения

Укажите уникальный URI, который AD можно использовать для обозначения этого приложения, либо оставьте пустым для Visual Studio создаст один.
