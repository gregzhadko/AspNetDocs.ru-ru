## <a name="overview-of-web-application-project-creation"></a>Обзор создания проекта веб-приложений

Следующие шаги показывают, как создать веб-проект:

1. Нажмите **новый проект** на странице **«Старт»** или в меню **«Файл».**
2. В диалоге **нового проекта** щелкните **web** в левом стеле и **ASP.NET web-приложению** в средней панели.

    ![Диалоговое окно "Новый проект"](creating-web-projects-in-visual-studio/_static/image1.png)

    Для создания [облачной службы Azure,](https://docs.microsoft.com/azure/cloud-services/cloud-services-how-to-create-deploy) [mobile service Azure](https://msdn.microsoft.com/library/windows/apps/dn629482.aspx)или [Azure WebJob](https://azure.microsoft.com/documentation/articles/web-sites-dotnet-deploy-webjobs)можно выбрать **облако** в левом стежке. Эта тема не охватывает эти шаблоны.
3. В правом стеле щелкните **Добавление приложений Insights к** флажке Project, если требуется мониторинг работоспособности и использования приложения. Дополнительные сведения см. в разделе о [мониторинге производительности в веб-приложениях](https://azure.microsoft.com/documentation/articles/app-insights-web-monitor-performance/).
4. Укажите **название**проекта, **местоположение**и другие параметры, а затем нажмите **OK.**

    Откроется диалоговое окно **Новый проект ASP.NET**.

    ![Диалоговое окно "Новый проект"](creating-web-projects-in-visual-studio/_static/image2.png)
5. Нажмите на шаблон.

    ![Выберите шаблон](creating-web-projects-in-visual-studio/_static/image3.png)
6. Если вы хотите добавить поддержку дополнительных фреймворков, не включенных в шаблон, нажмите на соответствующий флажок. (В показанном примере можно добавить MVC и/или Web API в проект Web Forms.)

    ![Добавление фреймворков](creating-web-projects-in-visual-studio/_static/image4.png)
7. <a id="testproj"></a>Если вы хотите добавить модульный тестовый проект, нажмите **«Добавить модульные тесты».**

    ![Добавление модульных тестов](creating-web-projects-in-visual-studio/_static/image5.png)
8. Если требуется другой метод проверки подлинности, чем то, что предоставляет шаблон по умолчанию, нажмите **Изменение аутентификации.**

    ![Кнопка настройки аутентификации](creating-web-projects-in-visual-studio/_static/image6.png)

    ![Настройка диалога аутентификации](creating-web-projects-in-visual-studio/_static/image7.png)

<a id="wf"></a>
### <a name="web-forms-template"></a>Шаблон веб-форм

Платформа Web Forms предоставляет следующие функции, которые позволяют быстро создавать веб-сайты, богатые функциями uI и доступа к данным:

- Дизайнер WYSIWYG в Visual Studio.
- Управление сервером, которое отображает HTML и можно настроить, задавая свойства и стили.
- Богатый ассортимент элементов управления для доступа к данным и отображения данных.
- Модель событий, которая предоставляет события, которые вы можете запрограммировать, как вы бы запрограммировать клиентское приложение, такие как WPF.
- Автоматическое сохранение состояния (данных) между запросами HTTP.

Как правило, создание приложения Web Forms требует меньше усилий по программированию, чем создание одного и того же приложения с помощью ASP.NET платформы MVC. Однако Web Forms предназначеннее не только для быстрой разработки приложений. Есть много сложных коммерческих приложений и инфраструктур, построенных поверх web-форм.

Поскольку страница Web Forms и элементы управления на странице автоматически генерируют большую часть разметки, отправленной в браузер, у вас нет такого мелкозернистого контроля над HTML, который ASP.NET предложенияm MVC. Декларативная модель настройки страниц и элементов управления сводит к минимуму количество кода, который нужно написать, но скрывает некоторое поведение HTML и HTTP. Например, не всегда возможно точно определить, какая именно разметка может быть сгенерирована элементом управления.

Платформа Web Forms не поддается так же легко, как ASP.NET MVC к шаблонам на основе методов разработки, таких как [тест-ориентированная разработка,](http://en.wikipedia.org/wiki/Test-driven_development) [разделение проблем,](http://en.wikipedia.org/wiki/Separation_of_concerns) [инверсия контроля,](http://en.wikipedia.org/wiki/Inversion_of_control)и [инъекция зависимости.](http://en.wikipedia.org/wiki/Dependency_injection) Если вы хотите написать код с фактором таким образом, вы можете; это просто не так автоматически, как в ASP.NET MVC рамках. Проект [ASP.NET Web Forms MVP](http://webformsmvp.com/) показывает подход, который облегчает разделение проблем и тестируемости при сохранении быстрой разработки, что веб-формы была построена для доставки. Microsoft SharePoint построен на веб-формах MVP.

Шаблон Web Forms создает образец приложения Web Forms, которое использует [Bootstrap](#bootstrap) для обеспечения адаптивного дизайна и тематических функций. На следующей иллюстрации показана главная страница.

<a id="mvc"></a>
### <a name="mvc-template"></a>Шаблон MVC

ASP.NET MVC был разработан для облегчения практики разработки на основе шаблонов, таких как [разработка на основе тестов,](http://en.wikipedia.org/wiki/Test-driven_development) [разделение проблем,](http://en.wikipedia.org/wiki/Separation_of_concerns) [инверсия контроля](http://en.wikipedia.org/wiki/Inversion_of_control)и [инъекция зависимости.](http://en.wikipedia.org/wiki/Dependency_injection) Платформа поощряет отделение слоя бизнес-логики веб-приложения от уровня презентации. Разделив приложение на модели (M), представления (V) и контроллеры (C), ASP.NET MVC может облегчить управление сложностью в больших приложениях.

С ASP.NET MVC вы работаете более непосредственно с HTML и HTTP, чем в веб-формах. Например, Web Forms могут автоматически сохранять состояние между запросами HTTP, но это необходимо четко закодировать в MVC. Преимущество модели MVC заключается в том, что она позволяет полностью контролировать то, что делает ваше приложение и как оно ведет себя в веб-среде. Недостатком является то, что вы должны написать больше кода.

MVC был разработан, чтобы быть расширяемым, предоставляя разработчикам энергии возможность настроить платформу для своих потребностей приложения. Кроме того, исходный код ASP.NET MVC доступен по лицензии OSI.

Шаблон MVC создает образец приложения MVC 5, который использует [Bootstrap](#bootstrap) для обеспечения адаптивного дизайна и тематических функций. На следующей иллюстрации показана главная страница.

## <a name="adding-support-for-additional-frameworks"></a>Добавление поддержки дополнительных рамок

При выборе шаблона автоматически выбирается флажок для платформы (ы), используемой шаблоном. Например, при выборе шаблона **Web Forms** выбирается флажок **Web Forms,** который не может его очистить.

![Выберите шаблон](creating-web-projects-in-visual-studio/_static/image21.png)

![Добавление фреймворков](creating-web-projects-in-visual-studio/_static/image22.png)

Можно выбрать флажок для платформы, которая не включена в шаблон, чтобы добавить поддержку этой платформы при создании проекта. Например, чтобы включить страницы Web Forms *.aspx* при выборе шаблона MVC, выберите флажок **Web Forms.** Или включить MVC при использовании шаблона Web Forms, нажмите на флажок **MVC.** Добавление фреймворка обеспечивает как время проектирования, так и поддержку времени выполнения. Например, если вы добавите поддержку MVC в проект Web Forms, вы сможете создать контроллеры и представления под мостом.

Если объединить web-формы и MVC в проекте и включить [дружественные URL-адреса](http://www.hanselman.com/blog/IntroducingASPNETFriendlyUrlsCleanerURLsEasierRoutingAndMobileViewsForASPNETWebForms.aspx) в веб-формах, могут возникнуть неожиданные проблемы с раской, когда один URL имеет несколько возможных целей. Маршруты, которые определены первыми, будут иметь приоритет. Например, если у `Home` вас есть контроллер и страница `http://contoso.com/home` *Home.aspx,* URL будет переходить на *Home.aspx,* если вы позвоните `EnableFriendlyUrls` по `MapRoute` методу, прежде `MapRoute` чем вызвать метод в *RouteConfig.cs,* или тот же URL будет идти в представление по умолчанию для контроллера, `Home` если вы звоните раньше. `EnableFriendlyUrls`

Добавление фреймворка не добавляет никакой функциональности образца приложения. Например, если вы добавляете поддержку Web Forms при выборе шаблона MVC, файл домашней страницы *Default.aspx* не создается. Добавляются только папки, файлы и ссылки, необходимые для поддержки платформы. По этой причине добавление инфраструктур не изменяет параметры аутентификации, которые реализуются кодом в образцах приложений, созданных шаблонами. Например, если вы выберете шаблон Empty и добавите web-формы или поддержку MVC, кнопка **настройки аутентификации** по-прежнему будет отключена.

В следующих разделах кратко описывается эффект каждого флажка.

### <a name="add-web-forms-support"></a>Добавление поддержки веб-форм

Создает *пустые папки\_данных и* *моделей* приложений и файл *Global.asax.* Они уже созданы всеми шаблонами, кроме шаблона Empty, поэтому выбор флажка Web Forms не имеет никакого значения для других шаблонов.

Шаблон Web Forms позволяет дружественные URL-адреса по умолчанию, но при добавлении поддержки Web Forms в другие шаблоны, выбрав контрольно-пропускной ящик Web Forms Friendly URL-адреса не будут автоматически включены.

### <a name="add-mvc-support"></a>Добавление поддержки MVC

Устанавливает пакеты MVC, Razor и WebPages NuGet, создает пустые папки *\_App Data,* *контроллеры,* *модели*и *просмотры,* создает папку *\_App Start* с *RouteConfig.cs* файлом и создает файл *Global.asax.*

### <a name="add-web-api-support"></a>Добавление поддержки Web API

Устанавливает пакеты WebApi и Newtonsoft.Json NuGet, создает пустые *папки* *\_App Data,* контроллеры и *модели,* создает папку *App\_Start* с *WebApiConfig.cs* файлом и создает файл *Global.asax.*

Если вы выберете **«Нет аутентификации»,** пример приложения не будет содержать веб-страниц для входа в систему, ни единого файла, указывающего, кто входит в систему, нет классов сущности для базы данных членства и строки подключения для базы данных членства.

<a id="indauth"></a>
### <a name="individual-user-accounts"></a>Учетные записи индивидуальных пользователей

При выборе **индивидуальных учетных записей пользователей**пример приложения будет настроен для использования ASP.NET Identity (ранее известного как ASP.NET членство) для проверки подлинности пользователей. ASP.NET Identity позволяет пользователю зарегистрировать учетную запись, создав имя пользователя и пароль на сайте или вомне в социальных сетях, такими как Facebook, Google, Microsoft Account или Twitter. Хранилище данных по умолчанию для профилей пользователей в ASP.NET Identity — это база данных Локального DB сервера S'L Server, которую можно развернуть в базу данных S'L Server или Azure S'L для производственной площадки.

В Visual Studio 2013 эти функции такие же, как и в Visual Studio 2012, но базовый код для ASP.NET системы членства был переписан. Преимущества новой базы кода включают в себя следующие:

- Новая система членства основана на [OWIN,](http://owin.org/) а не на модуле аутентификации ASP.NET форм. Это означает, что вы можете использовать тот же механизм проверки подлинности, независимо от того, используете ли вы веб-формы или MVC в IIS, или вы самостоятельно хостинг Web API или SignalR.
- Новая база данных о членстве управляется системным кодом Entity Framework Code First, и все таблицы представлены классами сущностей, которые можно изменить. Это означает, что вы можете легко настроить схему базы данных и связанный с профилем веб-ми-файлы в соответствии с вашими собственными потребностями, и вы можете легко развернуть обновления с помощью Code First Migrations.

Новая система членства внедряется автоматически в новые шаблоны, и она может быть реализована вручную в любом проекте, который нацелен на .NET 4.5 или позже.

ASP.NET Identity является хорошим выбором, если вы создаете веб-сайт в Интернете, который в основном для внешних клиентов. Если ваша организация использует Active Directory или Office 365 и вы хотите создать проект, который позволит единым вподписании для сотрудников и деловых партнеров, вариант **Организационных счетов** может быть лучшим выбором.

<a id="winauth"></a>
### <a name="windows-authentication"></a>Проверка подлинности Windows

Если вы выберете **аутентификацию Windows,** пример приложения будет настроен для использования модуля IIS подлинности Windows для проверки подлинности. Приложение будет отображать домен и идентификатор пользователя Active каталога или локальную учетную запись машины, которая входит в Windows, но не будет включать в себя регистрацию пользователей или пользовательский интерфейс. Эта опция предназначена для веб-сайтов Intranet.

Кроме того, можно создать сайт Intranet, который использует аутентификацию AD, выбрав [опцию On-Premises под организационными учетными записями.](#orgauthonprem) Опция On-Premises использует Windows Identity Foundation (WIF) вместо модуля аутентификации Windows. Для настройки опции On-Premises требуются дополнительные шаги, но WIF предоставляет функции, которые недоступны с модулем аутентификации Windows. Например, с помощью WIF можно настроить доступ к приложениям в Active Directory и данные каталога запросов.

<a id="orgauthonprem"></a>
### <a name="on-premises-organizational-authentication"></a>Организационная аутентификация на территории

![На территории организационной аутентификации](creating-web-projects-in-visual-studio/_static/image26.png)

Выберите эту опцию, если требуется включить аутентификацию учетных записей пользователей, определенных в Active Directory (AD) Windows Server, и не хотите использовать Azure AD. Эту опцию можно использовать для создания сайта Intranet или интернет-сайта. Для Интернет-сайта используйте службы Active Directory Federation (ADFS) для предоставления доступа к AD. Для получения дополнительной информации, [ASP.NET см.](http://www.cloudidentity.com/blog/2014/02/12/use-the-on-premises-organizational-authentication-option-adfs-with-asp-net-in-visual-studio-2013/)

Для сайта Intranet в качестве альтернативы вы можете выбрать [Windows Authentication](#winauth) вместо этой опции. Для опции аутентификации Windows не нужно предоставлять URL-адрес документа метаданных. Однако проверка подлинности Windows не дает вам возможности контролировать доступ к приложениям в Active Directory или данные каталога запросов.

#### <a name="on-premises-authority"></a>Орган по вопросам натерритории

Введите URL-адрес, упомянение на документ метаданных. Документ метаданных содержит координаты органа. Приложение будет использовать эти координаты для управления потоком веб-подписей.

#### <a name="application-id-uri"></a>Приложение ID URI

Предоставьте уникальный URI, который AD может использовать для идентификации этого приложения, или оставьте пустым, чтобы позволить Visual Studio создать его.
